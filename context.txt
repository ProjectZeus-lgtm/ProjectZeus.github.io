## Plan: SvelteKit Adventure Platform — Core Architecture

**TL;DR**: Stand up a SvelteKit app deployed on Vercel, with file-based routing, Google + Discord OAuth (via Arctic + custom session management), Turso (hosted SQLite) through Drizzle ORM, and SvelteKit API routes for auth, lobby, and AI proxy. The existing Azgaar engine and world-context pipeline stay untouched in src. The site lives in a new `site/` directory with its own package.json. The current ProjectZeus.github.io/ stays as-is (legacy world viewer) until the new app is ready to replace it. Six screens: Landing → Login → Adventures Dashboard → Create Adventure → Lobby → Adventure Screen (TBD). Multiplayer lobby uses short-polling for MVP with a clear upgrade path to WebSockets/PartyKit later.

---

### Stack Summary

| Layer | Choice | Why |
|-------|--------|-----|
| Framework | **SvelteKit (Svelte 5)** | File-based routing, server routes, Vercel adapter, tiny bundles |
| Hosting | **Vercel** | Free tier, auto-deploy from GitHub, serverless functions built-in |
| Database | **Turso** (hosted libSQL/SQLite) + **Drizzle ORM** | Works with serverless (no persistent filesystem needed), free tier, migrations, type-safe |
| Auth | **Arctic** (OAuth) + cookie sessions | Lightweight, framework-agnostic. Lucia is deprecated; Arctic is its successor for OAuth flows |
| AI Proxy | **SvelteKit `+server.ts` route** | Proxies OpenAI calls server-side so the API key never hits the browser |
| Real-time (lobby) | **Short-polling** (MVP) | Simple, no extra infra. Upgrade to PartyKit/WebSockets when needed |
| Styling | **CSS custom properties** (carry over existing design tokens from styles.css) | Keep the dark glassmorphism aesthetic, no CSS framework needed |

### Database Schema (Drizzle)

**`users`** table:

| Column | Type | Notes |
|--------|------|-------|
| `id` | text (ULID) | Primary key |
| `google_id` | text, nullable, unique | From Google OAuth |
| `discord_id` | text, nullable, unique | From Discord OAuth |
| `username` | text | Display name (from OAuth profile) |
| `avatar_url` | text, nullable | Profile picture |
| `created_at` | integer (unix ms) | |

**`sessions`** table:

| Column | Type | Notes |
|--------|------|-------|
| `id` | text | Primary key (random token) |
| `user_id` | text | FK → users |
| `expires_at` | integer (unix ms) | 30-day expiry |

**`adventures`** table:

| Column | Type | Notes |
|--------|------|-------|
| `id` | text (ULID) | Primary key |
| `name` | text | User-provided adventure name |
| `owner_id` | text | FK → users |
| `world_seed` | text, nullable | Azgaar seed if generated |
| `mode` | text | `'solo'` or `'multiplayer'` |
| `status` | text | `'lobby'` → `'active'` → `'completed'` |
| `created_at` | integer | |
| `updated_at` | integer | |

**`adventure_members`** table:

| Column | Type | Notes |
|--------|------|-------|
| `adventure_id` | text | FK → adventures |
| `user_id` | text | FK → users |
| `role` | text | `'owner'` or `'player'` |
| `is_ready` | integer (0/1) | Lobby ready-up flag |
| `joined_at` | integer | |

**`adventure_state`** table:

| Column | Type | Notes |
|--------|------|-------|
| `adventure_id` | text | FK → adventures, unique |
| `state_json` | text | Serialized game state (TBD shape) |
| `updated_at` | integer | |

### Steps

**Phase 0 — Scaffold**

1. Create `site/` directory. Run `npx sv create` with Svelte 5, TypeScript, SvelteKit. Install the Vercel adapter (`@sveltejs/adapter-vercel`).
2. Install dependencies: `drizzle-orm`, `@libsql/client` (Turso driver), `drizzle-kit`, `arctic`, `oslo` (crypto utils for sessions), `ulid`.
3. Create a Turso database (free tier) via `turso db create project-zeus`. Get the URL + auth token. Store in .env as `TURSO_URL` and `TURSO_AUTH_TOKEN`.
4. Set up OAuth apps: create a Google Cloud OAuth 2.0 client and a Discord Developer Application. Store client IDs/secrets in .env.
5. Configure svelte.config.js with `adapter-vercel` and any path aliases (e.g., `$engine` → `../src/engine` if we need to import engine code later).

**Phase 1 — Database + Auth**

6. Define the Drizzle schema in `site/src/lib/server/db/schema.ts` with all five tables listed above.
7. Create `site/src/lib/server/db/client.ts` — initialize Turso client + Drizzle instance. Export `db`.
8. Run `drizzle-kit generate` + `drizzle-kit migrate` to push schema to Turso.
9. Implement `site/src/lib/server/auth/google.ts` and `discord.ts` — configure Arctic providers with client ID/secret and callback URLs.
10. Implement `site/src/lib/server/auth/sessions.ts` — functions: `createSession(userId)` → generates random token, inserts into `sessions`, returns Set-Cookie header; `validateSession(cookieToken)` → looks up session, checks expiry, returns user or null; `deleteSession(token)`.
11. Implement `site/src/hooks.server.ts` — on every request, read the session cookie, call `validateSession()`, attach `event.locals.user` (or null). This makes auth state available to all `+page.server.ts` load functions.
12. Build auth routes:
    - `site/src/routes/auth/login/+page.svelte` — two buttons (Google, Discord). Each links to the OAuth provider's authorize URL (generated by Arctic).
    - `site/src/routes/auth/callback/google/+server.ts` — exchanges code for tokens, extracts profile, upserts into `users`, calls `createSession()`, redirects to `/adventures`.
    - `site/src/routes/auth/callback/discord/+server.ts` — same flow for Discord.
    - `site/src/routes/auth/logout/+server.ts` — POST handler, deletes session, clears cookie, redirects to `/`.

**Phase 2 — Layout + Landing Page**

13. Port the design tokens from styles.css (CSS custom properties: `--bg`, `--surface`, `--accent`, `--accent-2`, `--text`, `--text-muted`, etc.) into `site/src/app.css`. Keep the dark theme and glassmorphism foundation.
14. Build `site/src/lib/components/Nav.svelte` — minimal top bar: placeholder logo/text on the left, user avatar + sign-out on the right (if authenticated), "Sign In" button otherwise. Data comes from the parent layout.
15. Build `site/src/lib/components/GlassPanel.svelte` — a reusable `<slot>`-based wrapper applying the `.panel-card` glassmorphism style. Use this everywhere.
16. Build `site/src/routes/+layout.svelte` — wraps all pages with `<Nav>`, passes user from `+layout.server.ts` data.
17. Build `site/src/routes/+layout.server.ts` — returns `{ user: event.locals.user }` so all pages have access.
18. Build `site/src/routes/+page.svelte` — landing page. Hero section with a tagline (placeholder copy), a brief pitch blurb, and a single "Begin Your Adventure" CTA button that links to `/auth/login` (or `/adventures` if already signed in). Keep it minimal and atmospheric — dark background, maybe a subtle gradient or world-art placeholder. This page should feel like opening a book.

**Phase 3 — Adventures Dashboard + Creation**

19. Build `site/src/routes/adventures/+page.server.ts` — protected: if no user, redirect to `/auth/login`. Otherwise, query `adventures` joined with `adventure_members` for this user. Return the list.
20. Build `site/src/routes/adventures/+page.svelte` — shows the user's adventures as `AdventureCard` components (name, mode, status, member count, created date). Empty state: "No adventures yet." A prominent "New Adventure" button links to `/adventures/new`.
21. Build `site/src/lib/components/AdventureCard.svelte` — glass panel showing adventure name, mode badge (solo/multiplayer), status badge (lobby/active/completed), and a "Continue" or "Join Lobby" action link.
22. Build `site/src/routes/adventures/new/+page.svelte` — form with: adventure name input, solo/multiplayer toggle, and a "Create" button. Multiplayer shows a note: "You'll get an invite link to share."
23. Build `site/src/routes/adventures/new/+page.server.ts` — form action: validate input, insert into `adventures` (status: `'lobby'` for multiplayer, `'active'` for solo), insert into `adventure_members` (role: `'owner'`, ready: true for solo). Redirect to `/adventures/[id]/lobby` for multiplayer, or `/adventures/[id]` for solo.

**Phase 4 — Multiplayer Lobby**

24. Build `site/src/routes/api/lobby/[id]/+server.ts` — GET handler: return lobby state (adventure info + members list + ready statuses). This is the polling endpoint.
25. Build `site/src/routes/api/lobby/[id]/join/+server.ts` — POST handler: insert user into `adventure_members` if not already present and adventure is in lobby status.
26. Build `site/src/routes/api/lobby/[id]/ready/+server.ts` — POST handler: toggle `is_ready` for the current user. If ALL members are now ready, update adventure status to `'active'`.
27. Build `site/src/routes/adventures/[id]/lobby/+page.server.ts` — load adventure + members. If adventure status is `'active'`, redirect to `/adventures/[id]`.
28. Build `site/src/routes/adventures/[id]/lobby/+page.svelte` — shows: adventure name, shareable invite link (just the URL), list of players with ready indicators (`LobbyPlayer` components), a "Ready Up" toggle button, and a status message ("Waiting for all players..."). Polls `GET /api/lobby/[id]` every 3 seconds via `setInterval` in `onMount`. When all ready and status flips to `'active'`, auto-navigates to `/adventures/[id]`.
29. Build `site/src/lib/components/LobbyPlayer.svelte` — avatar, username, ready/not-ready indicator.
30. Build `site/src/lib/stores/lobby.ts` — a Svelte store that manages polling interval, exposes `members`, `allReady`, `adventureStatus`. Cleans up interval on component destroy.

**Phase 5 — Adventure Screen (Stub)**

31. Build `site/src/routes/adventures/[id]/+page.server.ts` — load adventure + state. If status is `'lobby'`, redirect to lobby. If user is not a member, 403.
32. Build `site/src/routes/adventures/[id]/+page.svelte` — placeholder adventure screen. Show: adventure name, mode, member list, world seed, and a big "Adventure begins here — TBD" panel. This is the frontier; everything past this point is designed later.

**Phase 6 — Deploy + Verify**

33. Connect the GitHub repo to Vercel. Set the root directory to `site/`. Configure environment variables (Turso, Google OAuth, Discord OAuth, OpenAI key).
34. Push to main. Verify auto-deploy.
35. Verify the full flow end-to-end: landing → sign in (Google or Discord) → adventures dashboard → create solo adventure → see adventure screen. Then: create multiplayer adventure → share lobby link → second user joins → both ready up → auto-navigate to adventure screen.

- **Lobby**: Open lobby link in incognito (second user signs in) → appears in member list. Both click ready → both navigate to adventure screen.
- **DB**: Check Turso dashboard — tables populated, sessions have proper expiry, adventures and members created correctly.
- **Mobile**: Test on narrow viewport — nav collapses gracefully, lobby and dashboard are usable.

### Decisions

- **Arctic over Lucia**: Lucia Auth was deprecated (early 2025). Arctic is its OAuth successor; we pair it with manual cookie session management rather than a session library.
- **Turso over raw SQLite**: Vercel serverless functions have no persistent filesystem. Turso is hosted libSQL (SQLite-compatible) that works over HTTP — perfect for serverless. Free tier covers this scale easily.
- **Short-polling over WebSockets for lobby**: Vercel serverless doesn't support persistent WebSocket connections. 3-second polling is fine for a lobby of 2-6 players. Clear upgrade path to PartyKit when real-time demands grow.
- **Monorepo with `site/` subdirectory**: Keeps the existing engine code in src untouched. SvelteKit lives in `site/` with its own dependencies. Later, the engine can be imported via Vite aliases if we want to run world generation server-side.
- **No CSS framework**: The existing design language (dark theme, glassmorphism, Inter font, CSS custom properties) is clean and distinctive. We port the tokens and build on them with Svelte scoped styles + the `GlassPanel` component.
- **ULID for IDs**: Sortable, URL-safe, no collision risk. Better than UUIDs for adventure URLs.
